Section 01

Why RAP Exists

RAP is not just a new way to write OData services — it is a complete programming model that brings persistence, locking, authorisation, draft, and the save sequence under one framework. Each of these was previously left to the individual developer to implement consistently across projects. RAP makes them declarative.

You write logic only where the business actually needs something specific — validations, field derivations, and custom actions. Everything else is handled by the framework based on what you declare.

Coming from years of classic ABAP and CDS development, the first question RAP raises is not whether it works. The question is how the model maps to what you already know, and where the thinking has to change. That mapping is what this series is built around.

→ Note — Clean Core

RAP was built with SAP's Clean Core strategy as a core requirement. Custom development must not touch SAP standard objects and must stay stable across upgrades. RAP supports this by design — all custom objects live in the customer namespace, the framework owns the standard integration points, and service bindings are derived from CDS definitions and need to be republished only when you edit your own CDS views.

For any programme planning a move to SAP BTP or S/4HANA Cloud, new transactional development built today using classic SEGW or BAPI patterns will need to be rebuilt before that move. Starting with RAP now avoids that cost later.

Section 02

When to Use RAP

RAP is the right choice for all new transactional application development on S/4HANA and SAP BTP ABAP Environment. The decision of which type of RAP to use depends on the constraints in your project.

✦ Design Choice — Managed vs Unmanaged

Managed is always the starting point. Unmanaged is the right choice only when the save process cannot be handed to the framework — usually because an existing BAPI has years of business logic that cannot be replicated in the project timeline. Choosing unmanaged for convenience means losing framework-provided ETag handling and the structured save sequence. Make this decision early and document why. It is difficult to change mid-project.

Section 03

RAP vs Classic ABAP OData

Developers who have built services through SEGW will immediately see what RAP removes. This is not just a syntax change — the way you think about building a service changes fundamentally.

CapabilityClassic SEGW + BAPIRAP
Service definitionManual in SEGW — changes need regeneration and republishingDerived from CDS projection — metadata updates after republishing the service binding
CRUD implementationManual code in CREATE_ENTITY, UPDATE_ENTITY, GET_ENTITY methodsFramework handles saves in managed scenario — developer writes only validation and derivation logic
LockingManual enqueue calls — applied differently across developersDeclared with lock master in BDEF — framework handles automatically
Draft handlingCustom build per application — no standard patternFramework-provided when with draft is declared in BDEF
AuthorisationManual checks inside service methods — easy to missTwo layers: BDEF handles operation-level; DCL objects control data-level access
OData versionV2 only via SEGWOData V4 by default; V2 compatibility binding available
ETag / concurrencyManual build if required — often skippedDeclared with etag master — framework checks on every update
Clean CoreNot by designBuilt for Clean Core

What stands out most about this comparison is not any single feature — it is that RAP removes a whole category of decisions that classic development left to each developer. Locking, ETags, audit fields — in SEGW you either remembered them or you did not. In RAP they are declared, or the framework does not allow you to proceed.

⚠ Watch Out

The most common mistake when moving from SEGW to RAP is writing direct database calls inside the implementation class for every operation. In the managed scenario, the framework owns the save process. Writing MODIFY statements directly against the table bypasses the framework buffer and causes inconsistent behaviour that is hard to trace.

→ Note — Why the Interface / Consumption Split Matters

Keeping interface and consumption views separate means multiple applications can share the same business object while each presenting it differently. A layout change in one application cannot affect another because each owns its own consumption view.

The interface view, once stable, becomes a shared contract. Any change to it needs to be checked against all applications that project from it. This discipline is what keeps a multi-application project manageable over time — and it is the first thing that breaks down when teams are under pressure.

The Buffer-First Mindset

The most notable shift coming from classic ABAP is not the annotations or the BDEF syntax — it is changing the instinct of reaching for a SELECT when reading the entity you are currently modifying.

In RAP, every operation inside an implementation method writes to a transactional buffer. The framework saves this buffer to the database only during the save sequence. Within a transaction, changes in the buffer are not yet in the database.

Once you understand that the buffer exists to support draft and the structured save sequence, the pattern makes complete sense.

This matters most for validations. A SELECT on the same entity being modified will miss the buffer changes and return old data. However, for reading unrelated tables — configuration tables, other master data — a direct SELECT is perfectly valid.

⚠ Watch Out — Reading the Current Entity Inside Methods

When you need to read the entity you are currently working on inside a validation or determination, always use READ ENTITIES IN LOCAL MODE. This reads from the buffer and reflects the current transaction state. IN LOCAL MODE skips DCL access control checks — this is correct when operating inside the business object's own implementation. For unrelated tables, a direct SELECT is fine.

Section 06

End-to-End Runtime Flow

Not all layers are active on every request. What changes between a read, a create, and an update is which layers get involved and in what order. The key difference is between the interaction phase — where user changes collect in the buffer — and the save sequence — where the buffer is checked and written to the database.

Fiori Elements UI ↔ Service Binding (OData V4)
UI layout is driven entirely by OData metadata from CDS annotations. Service binding routes requests to the business object and publishes metadata.

→ Note — Why Draft Handling Matters

The separation of interaction phase from save sequence makes draft possible. When draft is enabled, user changes are saved to a separate draft table after each action — without touching the active data table. A user filling a long form can close the browser, return the next day, and continue where they left off. The final validation and save to the active table happen only when the user confirms. Without draft, unsaved data is lost on session timeout — a real business problem for complex data entry forms.

      valid_to    : abap.dats;

✦ Design Choice — abap.utclong for Timestamps

The created_at and changed_at fields must be typed as abap.utclong. The framework needs a UTC timestamp for the etag master field precise enough to distinguish changes made by multiple users at the same time from different time zones. A date field is not precise enough. This is a small decision that is easy to get wrong the first time — and worth getting right from the start because changing a field type after data exists in the table is a migration, not a simple fix.

→ Note — Two Separate Authorisation Layers

The authorization master in the BDEF and the DCL access control object are two separate things. BDEF authorisation controls what operations a user can do — create, update, delete. Without DCL, any logged-in user can read all records through the OData service regardless of their authorisation profile.

Section 01

Behaviour Definition — Structure & Design

The Behaviour Definition (BDEF) is the transactional contract of a RAP business object. It does not contain ABAP code — it is a declarative specification of what the object can do, how it is locked, how concurrency is controlled, and what logic runs at each point in the transaction lifecycle.

There are two BDEFs for every RAP business object. The interface BDEF carries the full definition. The consumption BDEF is a short projection that delegates to it and controls which capabilities the service layer can use. This mirrors the interface/consumption split in the CDS layer.

✦ Design Choice — Managed vs Unmanaged Declaration

The first line of the interface BDEF determines the scenario type. managed implementation in class zbp_i_tst_plant_md unique means the framework owns persistence. unmanaged implementation in class zbp_i_tst_plant_md unique means you own persistence — the framework will call your implementation for the save sequence and you write the database operations yourself.

Managed is the right choice for new development. Unmanaged is reserved for wrapping legacy persistence logic that cannot be replaced within the project scope. This decision is made once at the start of the project — changing from unmanaged to managed later requires a full reimplementation of the save logic.

Section 02

Interface BDEF — Full Walkthrough

The interface BDEF for the Plant Master application covers the full range of declarations. Each keyword is a design decision, not just syntax.

ZI_TST_PLANT_MD — Interface Behaviour DefinitionBDEF · Managed
managed implementation in class zbp_i_tst_plant_md unique;
strict ( 2 );

define behavior for ZI_TST_PLANT_MD alias Plant
  persistent table ztst_plant_md
  lock master
  authorization master ( instance )
  etag master ChangedAt
{
  create ( authorization : global );
  update;
  delete;

  field ( readonly ) CreatedBy, CreatedAt;
  field ( readonly : update ) Plant;

  validation ValidatePlantKey       on save { create; }
  validation ValidateRequiredFields on save { create; update; }

  determination SetDefaultValues    on modify { create; }

  mapping for ztst_plant_md {
    Plant         = plant;
    PlantName     = plant_name;
    Country       = country;
    Region        = region;
    PlantType     = plant_type;
    Currency      = currency;
    ValidFrom     = valid_from;
    ValidTo       = valid_to;
    CreatedBy     = created_by;
    CreatedAt     = created_at;
    ChangedBy     = changed_by;
    ChangedAt     = changed_at;
  }
}

Keyword by Keyword

→ strict ( 2 )

This enables strict mode level 2 — the recommended setting for new RAP development. It enforces stricter syntax rules and prevents deprecated patterns. Always include this in new BDEFs. Older RAP implementations may use strict ( 1 ) or no strict mode at all — this is one of the first things to check when reviewing existing RAP code.

✦ Design Choice — persistent table

In the managed scenario, this is the only place you name the physical database table. The framework uses this together with the mapping block to translate between CDS element names (CamelCase) and table field names (underscore). If a field is missing from the mapping block, the framework cannot persist it — the field disappears silently on save. Every field that needs to be saved must appear in the mapping.

✦ Design Choice — lock master

Declares that this entity manages its own lock. When a user opens a record for editing, the framework places a lock on that record. This prevents two users from editing the same record at the same time. In a parent-child model — for example, a header with line items — the parent declares lock master and the child declares lock dependent by _Parent. For a flat entity like Plant, lock master is always the right declaration.

✦ Design Choice — etag master ChangedAt

The ETag is a version marker on each record. When a user reads a record, they receive the current ETag value. When they try to save, the framework compares the ETag they send against the current value in the database. If another user saved in between, the ETags differ and the update is rejected — protecting against silent overwrite. ChangedAt typed as abap.utclong is the correct ETag field. It changes on every save and is precise enough to distinguish concurrent updates.

✦ Design Choice — authorization master ( instance ) and create ( authorization : global )

authorization master ( instance ) means authorisation is checked at the record level inside the implementation class. You implement a get_instance_authorizations method that checks whether the current user is allowed to update or delete each specific record — useful when different users own different records. The alternative is authorization master ( global ), which applies a blanket check regardless of which record is being accessed.

create ( authorization : global ) on the create operation means the authorisation check for creating a new record is a global check — not per-record, since no record exists yet to check against. This is the correct and only logical setting for create. You cannot check instance-level authorisation on a record that does not exist yet.

✦ Design Choice — field ( readonly ) and field ( readonly : update )

field ( readonly ) CreatedBy, CreatedAt means these fields can never be set by the user — the framework fills them automatically via the @Semantics.* annotations on the interface view. field ( readonly : update ) Plant means the Plant key field can be set on create but cannot be changed after the record exists. This is the standard pattern for business key fields — set once, never changed.

Section 03

Consumption BDEF — Delegation

The consumption BDEF is intentionally minimal. It delegates all behaviour to the interface BDEF and controls which capabilities are visible to the service layer.

ZC_TST_PLANT_MD — Consumption BDEFBDEF · Projection
projection;
strict ( 2 );

define behavior for ZC_TST_PLANT_MD alias Plant
{
  use create;
  use update;
  use delete;
}

The use keyword exposes capabilities from the interface BDEF to the OData service. If you want to prevent deletion through the Fiori UI while keeping the operation available at the interface level — for example, for background processing — simply remove use delete from the consumption BDEF. The interface BDEF does not change. The deletion capability still exists but is not reachable through the service.

→ Note — Validations and Determinations in the Consumption BDEF

Validations and determinations declared in the interface BDEF are automatically active — you do not need to use them in the consumption BDEF. They run as part of the save sequence regardless. The consumption BDEF only needs to explicitly use standard operations (create, update, delete) and actions. This is a common point of confusion for developers new to RAP.

Section 04

Behaviour Implementation Class

The behaviour implementation class is where ABAP code lives. It is generated by the framework using Ctrl+1 Quick Fix on the BDEF in ADT. The generation creates the global class and the local class stubs for all declared validations, determinations, and actions.

Validation Method — Example

lhc_Plant — ValidatePlantKeyValidation Method
METHOD validateplantkey.

    " Read from buffer — not from database
  READ ENTITIES OF zi_tst_plant_md IN LOCAL MODE
    ENTITY Plant
      FIELDS ( Plant )
      WITH CORRESPONDING #( keys )
    RESULT DATA(plants).

  LOOP AT plants INTO DATA(plant_data).

    IF plant_data-Plant IS INITIAL.
      APPEND VALUE #(
        %tky = plant_data-%tky
        %msg = new_message_with_text(
          severity = if_abap_behv_message=>severity-error
          text     = 'Plant key cannot be empty' )
      ) TO reported-Plant.
      APPEND VALUE #( %tky = plant_data-%tky ) TO failed-Plant.
    ENDIF.

  ENDLOOP.

ENDMETHOD.
→ Note — %tky, failed, and reported

%tky is the transactional key — it identifies a record within the RAP buffer. failed is the table you append to when a record fails validation — the framework uses this to block the save for that record. reported is where you add messages that are shown to the user in the Fiori UI. Both are implicitly available in every validation method — you do not declare them. These three structures are the standard output of every RAP validation.

Section 05

EML — Entity Manipulation Language

EML is the ABAP-native language for reading and modifying RAP business objects. It is used in two contexts: inside implementation methods (to read or modify the entity being processed), and from external ABAP programs that consume a RAP business object.

EML is an in-process language. It cannot be called remotely like an RFC-enabled function module. For remote consumption of a RAP business object, the OData service is the correct channel.

EML — Common PatternsEntity Manipulation Language
// Reading from the buffer inside an implementation method
READ ENTITIES OF zi_tst_plant_md IN LOCAL MODE
  ENTITY Plant
    ALL FIELDS WITH CORRESPONDING #( keys )
  RESULT DATA(plants).

// Creating a record from an external ABAP program
MODIFY ENTITIES OF zi_tst_plant_md
  ENTITY Plant
    CREATE FIELDS ( Plant PlantName Country PlantType )
    WITH VALUE #( ( Plant = 'PL10' PlantName = 'New Plant'
                    Country = 'DE'  PlantType = 'MF' ) )
  REPORTED DATA(reported)
  FAILED DATA(failed)
  MAPPED DATA(mapped).

// Commit — required when calling from outside a RAP context
COMMIT ENTITIES
  RESPONSE OF zi_tst_plant_md
  FAILED DATA(commit_failed)
  REPORTED DATA(commit_reported).
⚠ Watch Out — COMMIT ENTITIES vs COMMIT WORK

When calling EML from an external ABAP program, use COMMIT ENTITIES — not COMMIT WORK. COMMIT ENTITIES triggers the full RAP save sequence including all validations declared in the BDEF. COMMIT WORK bypasses the RAP save sequence and writes directly to the database, skipping validations entirely. This is one of the most impactful mistakes in RAP integration scenarios.

Section 06

Validations vs Determinations

Both are declared in the interface BDEF and implemented in the local handler class. They look similar in structure but serve completely different purposes and run at different points in the transaction lifecycle.

AspectValidationDetermination
PurposeCheck data — report errors and block save if data is invalidDerive data — calculate and set field values based on other fields
Triggeron save — runs during the save sequence, after all modifications are in the bufferon modify — runs during the interaction phase, after the framework processes all modifications in a round trip
OutputAppends to failed and reported — blocks save if failed is populatedModifies fields in the buffer via MODIFY ENTITIES
Can block save?YesNo — it sets values, never blocks
Typical useDuplicate key check, mandatory field check, date range validationDefault value population, derived field calculation, status derivation
BDEF — Validation and Determination DeclarationInterface BDEF
// Validation — runs on save, can block the operation
validation ValidatePlantKey       on save { create; }
validation ValidateRequiredFields on save { create; update; }

// Determination — runs on modify, sets field values
determination SetDefaultValues    on modify { create; }
⚠ Watch Out — Determination Triggering Loops

A determination that modifies a field will trigger another on modify event. If another determination is declared for that field, it will run again — potentially creating a loop. The framework has loop detection, but the safest approach is to be explicit about which fields trigger each determination and to avoid having determinations that trigger each other. Always check keys before reading to confirm the determination is actually needed for this specific change.

→ Note — Save Sequence Order

Understanding the order in which things run during a save helps you place logic correctly. The correct sequence is: user modifications collected in buffer → determinations triggered on modify run during the interaction phase → on explicit save, validations run against the buffer → if all validations pass, the saver flushes the buffer to the database. If a validation fails, the save is blocked and the user sees the error messages. Determinations do not run again during the save sequence — they run only during the interaction phase when fields are modified.

Section 07

Actions in RAP

Actions are non-standard operations — operations beyond create, update, and delete. They are the RAP equivalent of custom function modules or BAPI methods that perform a specific business operation on a record. Examples: approve a request, copy a record, activate a plant, trigger an output.

✦ Design Choice — features : instance

features : instance means the action's availability is controlled per record by the implementation class. You implement a get_instance_features method that checks the current state of each record and returns whether the action is enabled, disabled, or hidden. This is how you show the "Activate Plant" button only for plants that are currently inactive — the button state reflects the record state automatically without any custom UI code.

Section 08

Draft Handling

Draft is the mechanism that allows a user to start filling a form, save their progress without committing to the active database, and return later to complete it. It is declared in the BDEF with a single addition and the framework generates the draft table and handles all draft-to-active transitions automatically.

BDEF — Draft DeclarationInterface BDEF · with draft
managed implementation in class zbp_i_tst_plant_md unique;
strict ( 2 );
with draft;   // This single addition enables the full draft framework

define behavior for ZI_TST_PLANT_MD alias Plant
  persistent table    ztst_plant_md
  draft table         ztst_plant_md_d   // Framework generates this table
  lock master
  total etag          LocalLastChangedAt
  etag master         ChangedAt
  authorization master ( instance )
{
  create;
  update;
  delete;
  draft action Edit;
  draft action Activate;
  draft action Discard;
  draft action Resume;
  draft determine action Prepare;
}
Draft ActionWhat It Does
EditCreates a draft copy of an active record — user edits the draft, not the active record
ActivateRuns all validations and saves the draft to the active table — the formal "Save" operation
DiscardDeletes the draft without saving — user abandons their changes
ResumeNavigates a user back to their existing draft when they return to a record that has an unsaved draft. Triggered by the framework on navigation — not a button the user clicks directly
PrepareRuns validations without saving — allows the UI to show errors before the user clicks Activate
→ Note — total etag and LocalLastChangedAt

When draft is enabled, a second ETag field total etag LocalLastChangedAt is required in addition to etag master ChangedAt. The etag master tracks changes to the active record. The total etag tracks the overall state including drafts — it changes when either the active record or a draft is modified. Both fields must be present as abap.utclong in the database table. LocalLastChangedAt is the SAP-recommended field name for the total ETag field — add it to your table definition alongside changed_at.

✦ Design Choice — When to Use Draft

Draft adds a draft table, additional OData operations, and a more complex UI flow. For simple master data forms where users typically fill and save in one session, draft is not always needed. Use draft when: the form has many fields and session timeouts are a real risk, when the creation process spans multiple steps or requires approvals before activation, or when multiple users may work on the same record at different times. For the Plant Master example in this series, draft is demonstrated as a declaration pattern — the business case for it depends on your specific workflow.

Section 01

Value Help — Three Design Patterns

Value help in RAP is defined in the consumption view using @Consumption.valueHelpDefinition. There are three distinct patterns depending on the source of the values. Choosing the right pattern matters — using the wrong one either adds unnecessary complexity or produces a value help with no data.

Pattern 1 — Domain-Based Value Help

When a field is typed with a custom data element that has a domain with fixed values, the value help is built as a CDS view that reads domain values from the database. This is the right pattern when your value set is small, stable, and maintained in the DDIC — for example, plant types: MF (Manufacturing), WH (Warehouse), OF (Office).

→ Note — Build a Common Domain Value Help

A clean reusable approach is to create one common domain value help view — ZI_DOMAIN — that reads domain values from the standard SAP domain values table. The view accepts the domain name as a parameter and is filterable by domain name. This single view can then be reused across all domain-based fields in any application — you only build the pattern once and reference it with a different domain name each time.

ZI_TST_PLANT_TYPE_VH — Domain-Based Value Help ViewValue Help CDS
@EndUserText.label: 'Plant Type Value Help'
@ObjectModel.dataCategory: #VALUE_HELP

define view entity ZI_TST_PLANT_TYPE_VH
  as select from ZI_DOMAIN( DomainName: 'ZTST_PLANT_TYPE' )
{
  key DomainValue      as PlantType,
      ShortDescription  as ShortDescription
}
Consumption View — Binding for Plant Type@Consumption.valueHelpDefinition
@Consumption.valueHelpDefinition: [{
  entity: {
    name:    'ZI_TST_PLANT_TYPE_VH',
    element: 'PlantType'
  }
}]
PlantType,
→ Note — @ObjectModel.dataCategory : #VALUE_HELP

This annotation marks the CDS view as a value help provider. Without it, the view can be used as a value help technically, but the annotation makes the intent explicit and enables better tooling support in ADT. Always add it to dedicated value help CDS views.

Pattern 2 — SAP Standard Value Help (I_* Views)

For standard SAP domains — currency, language, country — SAP provides published interface views in the I_* namespace. You do not build your own VH view. You point directly to the SAP-delivered view. This is the correct and upgrade-safe approach for standard fields.

Consumption View — Standard SAP Value HelpsSAP Standard I_* Views
// Currency — SAP standard value help
@Consumption.valueHelpDefinition: [{
  entity: { name: 'I_Currency', element: 'Currency' }
}]
Currency,

// Language — SAP standard value help
@Consumption.valueHelpDefinition: [{
  entity: { name: 'I_Language', element: 'Language' }
}]
Language,

// Country — SAP standard value help
@Consumption.valueHelpDefinition: [{
  entity: { name: 'I_Country', element: 'Country' }
}]
Country,
✦ Design Choice — Always Use I_* Views for Standard Fields

Building a custom VH view for currency, language, or country is unnecessary work and introduces maintenance risk. SAP-delivered I_* views are kept up to date with every release. More importantly, these views are part of SAP's released C1 API surface — they are stable across S/4HANA upgrades. Custom replacements are not. Use the standard where it exists.

Pattern 3 — CDS-Based Value Help from a Custom Entity

When the value set comes from a custom database table — not a domain and not an SAP standard — you build a dedicated CDS view that selects from that table and expose it as a value help.

ZI_TST_PLNT_VH — Custom Entity Value HelpValue Help CDS
@ObjectModel.dataCategory: #VALUE_HELP

define view entity ZI_TST_PLNT_VH
  as select from ztst_plant_md
{
  key plant       as Plant,
      plant_name  as PlantName
}
⚠ Watch Out — Value Help View Must Be Exposed in Service Definition

The value help popup makes a separate OData call to load its list of values. For this call to work, the VH CDS view must be explicitly listed in the service definition. If you define @Consumption.valueHelpDefinition in the consumption view but forget to expose the VH view in the service definition, the popup opens but shows no data. This is one of the most common oversights in RAP development — always verify the service definition when a value help appears empty.

Section 02

Dependent Value Help with additionalBinding

The most useful value help pattern is a dependent value help — where the list shown in a popup is filtered based on another field already filled in the form. The Region field in the Plant Master application is a good example: when the user has already selected Country, the Region popup should show only regions for that country.

Consumption View — Region with additionalBindingDependent Value Help
@Consumption.valueHelpDefinition: [{
  entity: {
    name:    'I_Region',
    element: 'Region'
  },
  additionalBinding: [{
    localElement: 'Country',   // field in THIS consumption view
    element:      'Country',   // matching field in the VH entity
    usage:        #FILTER_AND_RESULT
  }]
}]
Region,
usage valueWhat it does
#FILTER_AND_RESULTPre-filters the popup using the local field value AND writes the selected value back to the local field. Use for parent-child lookups like Country → Region.
#FILTERPre-filters the popup only — does not write back. Use when you want to narrow the list but the user's selection does not need to update another field.
#RESULTWrites back the selected value to a local field — no pre-filtering. Use when you want to auto-fill a field based on what the user selects from the popup.
✓ Good Practice — additionalBinding for Parent-Child Fields

Apply this pattern wherever you have parent-child field relationships in your form: Country → Region, Company Code → Business Area, Plant → Storage Location. Without it, the popup shows all values globally and the user must know the exact code. With it, the form guides the user naturally. This is a small annotation that has a significant impact on usability — and it costs nothing to implement.

Section 03

Service Definition & Service Binding

The service definition and service binding are the final two artefacts that make a RAP business object accessible as an OData service. Without them, the CDS views and BDEFs exist in the system but nothing can call them.

ZSD_TST_PLANT_MAINT — Service DefinitionService Definition
@EndUserText.label: 'Plant Master Maintenance Service'

define service ZSD_TST_PLANT_MAINT {
  expose ZC_TST_PLANT_MD         as Plant;
  expose ZI_TST_PLANT_TYPE_VH    as PlantTypeValueHelp;
  // All value help views used in the consumption view must be listed here
}

Two service bindings are created for the same service definition — one for each OData version needed.

BindingTypeWhen to Use
ZSB_TST_PLANT_UI_V2OData V2 — UIADT Preview for Fiori Elements app — the quick test preview in Eclipse. V2 is required for this specific tool.
ZSB_TST_PLANT_UI_V4OData V4 — UIProduction Fiori Elements applications. OData V4 is the strategic version — it supports draft handling, richer query capabilities, and deep entity operations.
→ Note — Republish After Editing a CDS View

When you edit the structure of a CDS view — adding a field, changing an association, or adding annotations that affect the metadata — the service binding does not update automatically. You must republish the service binding in ADT for the changes to reflect in the OData metadata. This is a step that is easy to forget during development — if your Fiori app is not showing a new field you added to the CDS view, check whether the service binding has been republished.

→ Note — The TST Namespace Segment

The TST segment in the trial system names is a training package identifier — common in SAP learning environments. In a real project, the SAP convention is that the ZI_ and ZC_ prefixes already communicate the artefact type. Adding a namespace segment between the prefix and the object name — like ZI_MM_PLANT_MD — is common in large programmes to indicate the module area (MM, SD, FI), but the SAP standard does not require it. What matters most is that the convention is agreed and applied consistently across the entire project from day one.

Common Pitfalls in RAP Projects

These are the design and implementation mistakes that appear most often in RAP code reviews. Each one either breaks something immediately or creates a problem that surfaces later when it is more expensive to fix.

PitfallWhat Goes WrongHow to Avoid
UI annotations in the interface viewSecond application projecting from the same interface view inherits unintended UI behaviourAll @UI.* annotations belong in the consumption view or a Metadata Extension on the consumption view

Section 06

Unmanaged RAP — Coming in the Next Series

This series has focused entirely on the managed RAP scenario — where the framework handles database persistence and you declare behaviour rather than implement it from scratch. This is the right starting point and the right default for all new development.

There is a second scenario — unmanaged RAP — where you take full ownership of the save sequence. The framework still provides the OData service, the BDEF contract, and the structured save flow, but the actual database operations are written by you in the implementation class. This matters specifically when:

  • You are wrapping a legacy BAPI that has decades of business logic and cannot be replaced
  • The persistence layer is a complex custom structure that the managed framework cannot map directly
  • You need to call multiple BAPIs in sequence as part of a single save operation
